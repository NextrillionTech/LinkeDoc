## Summary

We will construct the initial MVP skeleton for LinkeDoc. This includes a decoupled architecture:
1.  **Backend API Server**: Node.js/Express with TypeScript and PostgreSQL (via Prisma). Manages credentials, pending admin verification queue, P2P connection logic, forums, and a free job listing board.
2.  **Frontend Client**: React + Vite with TypeScript and Vanilla CSS. Implements professional profile builder, specialty discussions, peer search/networking, and job search interface.

## Technical Context

**Language/Version**: Node.js v18+, TypeScript v5.0+

**Primary Dependencies**: Express, Prisma Client, Zod (validation), JWT, React Router v6

**Storage**: PostgreSQL

**Testing**: Jest + Supertest (Backend), Vitest + React Testing Library (Frontend)

**Target Platform**: Web Browsers (Mobile and Desktop responsive)

**Project Type**: web-service + frontend application

**Performance Goals**: < 200ms p95 response times for search filtering and profile updates.

**Constraints**: User registration for medical professionals enqueues them in "Pending Verification" state, blocking all network/forum features until manual admin approval. Flagged posts are hidden instantly from standard feeds.

**Scale/Scope**: Medical professionals (Doctors, Nurses, Pharmacists, Researchers), Recruiters, and Administrators.

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

- **Principle I: Peer-to-Peer Privacy & HIPAA Compliance**: Flagging API endpoint (`POST /api/forums/report`) enqueues flagged items and sets their state to `HIDDEN` instantly. Database query filters out any thread/reply with state `HIDDEN`. (Status: **PASSED**)
- **Principle II: API-First Modular Design**: API endpoints decoupled from frontend components and documented in separate API contracts under `contracts/`. (Status: **PASSED**)
- **Principle III: Test-Driven Development (TDD)**: Mock tests for connection requests, forum category retrieval, and auth status verification must pass before implementing DB models/handlers. (Status: **PASSED**)
- **Principle IV: Accessibility & Responsive UX**: Handled strictly via CSS styles, ensuring responsive layouts and high contrast. (Status: **PASSED**)
- **Principle V: Type-Safe API Contracts**: Shared type definitions mapping database models to client payloads, using Zod schemas for validation. (Status: **PASSED**)
